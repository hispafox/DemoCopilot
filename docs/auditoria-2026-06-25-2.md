### 🔴 [CRÍTICO] — Seguridad — Vulnerabilidad de alta gravedad en dependencia SQLite

**Fichero:** [AppTodoList.csproj](../AppTodoList.csproj) (línea 14)  
**Paquete afectado:** `SQLitePCLRaw.lib.e_sqlite3` versión 2.1.11 (dependencia transitiva de `Microsoft.EntityFrameworkCore.Sqlite`)  
**CVE:** [GHSA-2m69-gcr7-jv3q](https://github.com/advisories/GHSA-2m69-gcr7-jv3q)

**Descripción:**  
Durante la compilación, NuGet reporta:
```
warning NU1903: El paquete "SQLitePCLRaw.lib.e_sqlite3" 2.1.11 tiene una vulnerabilidad de gravedad alta conocida
```

Esta vulnerabilidad es transitiva — no se referencia directamente en el `.csproj`, sino que es traída por `Microsoft.EntityFrameworkCore.Sqlite` 10.0.9.

**Riesgo:**  
Una vulnerabilidad de **alta gravedad** en la capa de acceso a base de datos puede exponer la aplicación a:
- Explotación de memoria (buffer overflow, use-after-free)
- Ejecución arbitraria de código si se procesan datos maliciosos
- Denegación de servicio (DoS)
- Corrupción de base de datos

Aunque SQLite se ejecuta in-process (no es un servicio de red), esto sigue siendo un riesgo en escenarios donde datos de entrada no confiables llegan a queries SQL.

**Acción requerida:**
1. Verificar si existe una versión más reciente de `Microsoft.EntityFrameworkCore.Sqlite` que resuelva la dependencia a una versión parcheada de `SQLitePCLRaw.lib.e_sqlite3`.
2. Si no existe actualización oficial:
   - Forzar la dependencia explícita a una versión segura de `SQLitePCLRaw.lib.e_sqlite3` en el `.csproj`:
     ```xml
     <PackageReference Include="SQLitePCLRaw.lib.e_sqlite3" Version="[versión-segura]" />
     ```
   - Consultar el advisory en GitHub para conocer la versión mínima recomendada.
3. Ejecutar `dotnet restore` y `dotnet build` para verificar que la advertencia desaparece.
4. Considerar cambiar a PostgreSQL, SQL Server LocalDB o una alternativa si SQLite no es crítico para el proyecto.

### 🟡 [MEDIO] — EF Core — Falta `AsNoTracking()` en queries de solo lectura

**Ficheros afectados:**
- [LogicaNegocio/CategoriaLogica.cs](../LogicaNegocio/CategoriaLogica.cs) línea 16
- [LogicaNegocio/TodoLogica.cs](../LogicaNegocio/TodoLogica.cs) línea 14
- [LogicaNegocio/PlantillaLogica.cs](../LogicaNegocio/PlantillaLogica.cs) línea 16
- [LogicaNegocio/UsuarioAsignadoLogica.cs](../LogicaNegocio/UsuarioAsignadoLogica.cs) línea 16

**Descripción:**  
Los métodos `ObtenerTodosAsync()` (y variantes como `ObtenerTodasAsync()`) ejecutan queries de solo lectura sobre el `DbContext`, pero no utilizan `.AsNoTracking()`.

**Ejemplo en CategoriaLogica:**
```csharp
public async Task<IEnumerable<Categoria>> ObtenerTodosAsync()
    => await _contexto.Categorias.ToListAsync();
```

Debería ser:
```csharp
public async Task<IEnumerable<Categoria>> ObtenerTodosAsync()
    => await _contexto.Categorias.AsNoTracking().ToListAsync();
```

**Riesgo:**
- **Overhead de Change Tracker**: EF Core rastrea todas las entidades cargadas para detectar cambios, lo cual consume memoria y CPU innecesariamente en consultas que solo leen datos.
- **Escalabilidad**: en endpoints con tráfico alto (`GET /api/tareas`, `GET /api/categorias`), este overhead se multiplica linealmente con el número de peticiones.
- **Mejor práctica ignorada**: es una optimización simple y gratuita que debería aplicarse siempre en consultas de solo lectura.

**Por qué es MEDIO y no CRÍTICO:**  
En una aplicación pequeña como esta demo (pocas entidades, volumen bajo), el impacto es imperceptible. Pero es deuda técnica que se acumula y hará daño en cuanto el proyecto escale.

**Acción requerida:**
1. En **todos** los métodos de `LogicaNegocio` que solo leen datos (`ObtenerTodosAsync`, `ObtenerPorIdAsync` cuando no se modifica después), añadir `.AsNoTracking()` inmediatamente después del `DbSet<T>`.
2. Ejemplo corregido:
   ```csharp
   public async Task<IEnumerable<TodoItem>> ObtenerTodosAsync(int? categoriaId = null)
   {
       var query = _contexto.TodoItems
           .AsNoTracking()  // ← AÑADIR AQUÍ
           .Include(t => t.UsuarioAsignado)
           .Include(t => t.Categoria)
           .AsQueryable();

       if (categoriaId.HasValue)
           query = query.Where(t => t.CategoriaId == categoriaId.Value);

       return await query.ToListAsync();
   }
   ```
3. Aplicar el mismo cambio en `CategoriaLogica`, `PlantillaLogica`, `UsuarioAsignadoLogica`.

**Skill responsable:** `logica-negocio` (no incluye la instrucción de añadir `AsNoTracking()` automáticamente)

### 🟡 [MEDIO] — Validaciones — Faltan anotaciones `[Required]` y `[MaxLength]` en DTOs de entrada

**Descripción:**

**Ejemplo en CrearTareaDto:**

public class CrearTareaDto

    // ...

Comparar con `CrearCategoriaDto` (que SÍ tiene validaciones):
```csharp

    [MaxLength(100, ErrorMessage = "El nombre no puede superar 100 caracteres")]
    public string Nombre { get; set; } = string.Empty;

**Riesgo:**
- **Validación inconsistente**: `CategoriasDtos` y `UsuariosAsignadosDtos` tienen validaciones, pero `TareasDtos` y `PlantillasDtos` no. Esto crea una experiencia inconsistente de la API.

**Por qué es MEDIO y no ALTO:**  
El modelo de datos en `AppDbContext` tiene restricciones Fluent API (`HasMaxLength(200)`, `IsRequired()`), por lo que la base de datos rechazará datos inválidos. Sin embargo, esto genera excepciones de EF Core en lugar de respuestas `400 Bad Request` claras.

**Acción requerida:**
1. Añadir en `CrearTareaDto` y `ActualizarTareaDto`:
   ```csharp
   [Required(ErrorMessage = "El título es obligatorio")]
   [MaxLength(200, ErrorMessage = "El título no puede superar 200 caracteres")]
   public string Title { get; set; } = string.Empty;
   ```
2. Añadir en `CrearPlantillaDto` y `ActualizarPlantillaDto`:
   ```csharp
   [Required(ErrorMessage = "El título es obligatorio")]
   [MaxLength(200, ErrorMessage = "El título no puede superar 200 caracteres")]
   public string Titulo { get; set; } = string.Empty;
   ```
3. Verificar que el atributo `[ApiController]` está presente en todos los controladores (ya lo está) — esto activa la validación automática.

**Skill responsable:** `dto` (las instrucciones dicen "Solo se pueden usar anotaciones de validación básicas de ASP.NET si el análisis las especifica explícitamente", pero no define un comportamiento determinista)

### 🟡 [MEDIO] — EF Core — Include innecesario en `TodoLogica.ObtenerPorIdAsync` cuando se usa `FindAsync`

**Fichero:** [LogicaNegocio/TodoLogica.cs](../LogicaNegocio/TodoLogica.cs) línea 28

**Descripción:**  
El método `ObtenerPorIdAsync` carga la entidad con `Include`:
```csharp
public async Task<TodoItem?> ObtenerPorIdAsync(int id)
    => await _contexto.TodoItems
        .Include(t => t.UsuarioAsignado)
        .Include(t => t.Categoria)
        .FirstOrDefaultAsync(t => t.Id == id);
```

Sin embargo, hay inconsistencia con otros métodos de lógica:
- `CategoriaLogica.ObtenerPorIdAsync` usa `FindAsync(id)` (más eficiente, usa cache)
- `PlantillaLogica.ObtenerPorIdAsync` usa `FindAsync(id)`
- `UsuarioAsignadoLogica.ObtenerPorIdAsync` usa `FindAsync(id)`

**Problema:**  
`FindAsync` es más eficiente para búsquedas por PK porque:
1. Primero busca en el cache del ChangeTracker (no va a la base de datos si la entidad ya está tracked).
2. Genera SQL más simple (`SELECT * FROM TodoItems WHERE Id = @p0`).

Pero si necesitas cargar relaciones, **no puedes usar `FindAsync`** — tienes que usar `FirstOrDefaultAsync` con `Include`.

En este caso, `TodoLogica.ObtenerPorIdAsync` **sí necesita** `Include` porque el servicio mapea `UsuarioAsignadoNombre` y `CategoriaNombre` desde las navegaciones. Por tanto, **el código es correcto**.

Sin embargo, la **inconsistencia** es confusa: ¿por qué `TodoLogica` usa `FirstOrDefaultAsync + Include` y los demás usan `FindAsync`? La respuesta es que los demás no cargan relaciones, pero esto no es evidente al leer el código.

**Riesgo:**  
Si en el futuro `CategoriaLogica.ObtenerPorIdAsync` necesita cargar relaciones (por ejemplo, `Include(c => c.Tareas)`), un desarrollador podría pensar que puede seguir usando `FindAsync`, lo cual no funcionará.

**Por qué es MEDIO y no BAJO:**  
No es un bug, pero es un patrón inconsistente que puede llevar a bugs futuros.

**Acción requerida (opcional):**
1. Documentar con un comentario por qué `TodoLogica` usa `FirstOrDefaultAsync + Include`:
   ```csharp
   // Usamos FirstOrDefaultAsync + Include porque necesitamos cargar UsuarioAsignado y Categoria
   // para mapear UsuarioAsignadoNombre y CategoriaNombre en el servicio.
   // FindAsync no soporta Include.
   public async Task<TodoItem?> ObtenerPorIdAsync(int id)
       => await _contexto.TodoItems
           .Include(t => t.UsuarioAsignado)
           .Include(t => t.Categoria)
           .FirstOrDefaultAsync(t => t.Id == id);
   ```
2. O bien, añadir un parámetro `bool incluirRelaciones = false` y condicionar el `Include`:
   ```csharp
   public async Task<TodoItem?> ObtenerPorIdAsync(int id, bool incluirRelaciones = false)
   {
       if (!incluirRelaciones)
           return await _contexto.TodoItems.FindAsync(id);

       return await _contexto.TodoItems
           .Include(t => t.UsuarioAsignado)
           .Include(t => t.Categoria)
           .FirstOrDefaultAsync(t => t.Id == id);
   }
   ```

**Skill responsable:** `logica-negocio` (no documenta la diferencia entre `FindAsync` y `FirstOrDefaultAsync + Include`)

### 🟡 [MEDIO] — Skill `dto` — Falta regla determinista para validaciones automáticas

**Fichero:** [.github/skills/dto/SKILL.md](../.github/skills/dto/SKILL.md) línea 88

**Descripción:**  
El skill `dto` tiene esta instrucción:
> "Sin anotaciones de validación: la validación de dominio va en la lógica de negocio (`LogicaNegocio/`), no en los DTOs. Solo se pueden usar anotaciones de validación básicas de ASP.NET **si el análisis las especifica explícitamente**."

Este criterio es **ambiguo** y deja la decisión al LLM, causando inconsistencia:
- `CategoriasDtos.cs` tiene validaciones `[Required]`, `[MaxLength]`, `[RegularExpression]`.
- `UsuariosAsignadosDtos.cs` tiene validaciones `[Required]`, `[MaxLength]`, `[EmailAddress]`.
- `TareasDtos.cs` **no tiene validaciones** en `Title` (campo obligatorio).
- `PlantillasDtos.cs` **no tiene validaciones** en `Titulo` (campo obligatorio).

**Riesgo:**
- **Inconsistencia sistemática**: cada vez que se genera un DTO nuevo, la presencia o ausencia de validaciones depende de si el análisis las "especifica explícitamente" (criterio subjetivo).
- **Validaciones faltantes en producción**: sin `[Required]`, el backend acepta strings vacíos que luego fallan al intentar persistir.

**Mejora propuesta para el skill:**

Reemplazar la sección "Sin anotaciones de validación" por:

```markdown
#### Validaciones automáticas en DTOs de entrada

Para garantizar consistencia, aplicar **automáticamente** estas reglas:

1. **Toda propiedad `string` no nullable** (`string`, no `string?`) en un DTO de entrada (`CrearXxxDto`, `ActualizarXxxDto`) debe llevar:
   ```csharp
   [Required(ErrorMessage = "El {campo} es obligatorio")]
   [MaxLength(200, ErrorMessage = "El {campo} no puede superar 200 caracteres")]
   ```
   Ajustar el límite según el modelo de dominio si `HasMaxLength` en Fluent API especifica otro valor.

2. **Toda propiedad `string` nullable** (`string?`) debe llevar solo `[MaxLength]`, sin `[Required]`.

3. **Propiedades de tipo primitivo no nullable** (`int`, `bool`, `DateTime`):
   - No llevan `[Required]` (el compilador ya obliga a que tengan valor).
   - Si el análisis especifica rango válido, añadir `[Range(min, max)]`.

4. **Validaciones especializadas según tipo**:
   - Email: `[EmailAddress]`
   - Regex (formato hex, URL, etc.): `[RegularExpression(@"patrón")]`

Esta regla se aplica **por defecto**, sin necesidad de que el análisis lo especifique explícitamente. Si el análisis dice explícitamente que un campo **no** debe validarse (caso excepcional), entonces omitir la anotación.
```

**Impacto de aplicar la mejora:**  
Genera DTOs con validaciones básicas desde el inicio, reduciendo deuda técnica y mejorando robustez de la API automáticamente.

Para métodos que **solo leen datos** (`ObtenerTodosAsync`, `ObtenerPorIdAsync` sin modificación posterior), añadir **automáticamente** `.AsNoTracking()` inmediatamente después del `DbSet<T>`:

**Impacto de aplicar la mejora:**

**Línea:** ~60 (sección "Paso 2 — Verificar que los servicios usan DTOs")

**Descripción:** El skill tiene un paso de verificación que comprueba que los servicios usen DTOs, pero **no valida que el controlador solo inyecte servicios** (`IXxxService`), no `AppDbContext` ni `IXxxLogica` directamente.

En el código auditado, todos los controladores están correctos (solo inyectan servicios), pero **no porque el skill lo imponga**, sino porque los desarrolladores siguieron la convención. Si alguien intenta inyectar `AppDbContext` directamente, el skill no lo detectaría.

**Mejora propuesta:**

Añadir en la sección "Paso 2 — Verificar que los servicios usan DTOs", después del párrafo existente:

```markdown
### Verificación adicional: dependencias del controlador

Antes de generar el constructor del controlador, **verificar que todas las dependencias son interfaces de servicio** (`IXxxService`).

**RECHAZAR la generación** si se intenta inyectar:
- `AppDbContext` (la arquitectura requiere que el controlador solo hable con servicios)
- Clases de `LogicaNegocio` (`IXxxLogica`) — el controlador no debe saltarse la capa de servicio
- Clases de `Models` (entidades de dominio)
- Cualquier clase concreta (salvo `ILogger<T>` que es permitido)

**Mensaje de error:** "El controlador solo puede inyectar servicios (`IXxxService`) y `ILogger<T>`. Para acceder a datos, crea o usa un servicio existente."
```

**Por qué es BAJO y no MEDIO:**  
No hay violaciones de arquitectura en el código auditado, por lo que el skill funcionó bien en la práctica. Esta mejora es **preventiva** para futuros desarrollos.

**Impacto de aplicar la mejora:**  
Previene automáticamente violaciones de arquitectura de capas en futuros controladores generados por este skill.

#### 🔵 [BAJO] — Skill `servicio` no menciona mapeo bidireccional completo

**Fichero:** [.github/skills/servicio/SKILL.md](../.github/skills/servicio/SKILL.md)  
**Línea:** ~140 (sección "Estructura obligatoria de la implementación" → método `MapearADto`)

**Descripción:** El skill menciona mapeo pero no exige verificación de que **TODOS** los campos del DTO se mapeen desde la entidad.

En `PlantillaService.MapearTareaADto`, faltan 4 campos (`UsuarioAsignadoId`, `UsuarioAsignadoNombre`, `CategoriaId`, `CategoriaNombre`), lo cual genera DTOs incompletos.

**Mejora propuesta:**

Añadir en la sección "Estructura obligatoria de la implementación", después del método `MapearADto`:

```markdown
#### Checklist de mapeo completo

Antes de considerar terminado el método `MapearADto`, verificar que **todos** los campos del DTO de salida (`<Recurso>Dto`) están mapeados desde la entidad:

1. Abrir el fichero `Dtos/<Recurso>Dtos.cs` y listar todas las propiedades de `<Recurso>Dto`.
2. Para cada propiedad, verificar que existe una línea correspondiente en `MapearADto`:
   - Campos directos: `Id = entidad.Id`
   - Campos de navegación: `CategoriaNombre = entidad.Categoria?.Nombre`
3. Si un campo no se puede mapear porque la entidad no tiene esa información, documentar por qué con un comentario:
   ```csharp
   // CategoriaNombre no está disponible aquí porque la entidad no incluye la navegación
   CategoriaNombre = null
   ```

**Regla:** Nunca dejar campos del DTO sin asignar — el compilador no lo detecta si el DTO tiene `= string.Empty` por defecto.
```

**Por qué es BAJO:**  
Solo un caso detectado (`PlantillaService`), y no es grave (el DTO se devuelve, solo con campos faltantes).

**Impacto de aplicar la mejora:**  
Garantiza que DTOs de salida siempre están completos.
